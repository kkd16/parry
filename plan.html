What Chitin Is

Chitin is a security enforcement layer for AI agents and coding tools. It intercepts every tool call — shell commands, file edits, API requests, emails — classifies each by blast radius, and enforces your rules deterministically. It runs outside the agent, so it can't be prompt-injected, reasoned with, or turned off by a confused model.

It ships as a single Go binary with two modes:

Proxy Mode
Sits between any MCP-based agent and its tool servers. Intercepts all JSON-RPC traffic. For OpenClaw, NanoClaw, and any custom MCP agent.
Check Mode
Reads a tool call on stdin, returns allow/block. Plugs into Claude Code hooks, Cursor hooks, Copilot hooks, or any tool with pre-execution checks.
Same policy engine. Same scanner. Same rules. Different entry points.

Why It Matters

AI agents and coding tools can run shell commands, send emails, edit files, and make API calls on your behalf. When they go wrong — through prompt injection, context drift, or misconfiguration — consequences are real. A Meta researcher's agent mass-deleted her emails. Check Point found Claude Code configs could execute arbitrary commands before the trust dialog appeared. Cisco found 26% of agent skills contained vulnerabilities.

Most existing security tools run inside the agent as natural-language instructions. A prompt injection can override those. Your security skill gets talked out of protecting you.

Chitin runs out-of-process. It enforces deterministic rules. The agent doesn't know it's there. It can't be persuaded, overridden, or ignored.
Design Principles
01
ML signals, rules enforce
The ONNX scanner provides probabilities. The policy engine makes binary decisions. No LLM ever sits in the enforcement path.
02
Observe before enforce
New installs run in observe mode for 7 days. Build trust with data, not promises. Switch to enforcement when the user is ready.
03
Classify by consequence, not verb
The blast-radius taxonomy (T1–T5) classifies actions by what happens if they go wrong, not what they're called.
04
External enforcement
Chitin runs out-of-process. The agent cannot override, ignore, or reason around Chitin's decisions.
05
Local-first, always
All data, models, and enforcement stay on the user's machine. Nothing leaves. Non-negotiable.
06
Protocol-native, tool-agnostic
Proxy mode secures MCP. Check mode secures anything with pre-exec hooks. Same engine, universal reach.
07
Fail closed
If Chitin crashes, all tool calls are blocked. Silent failure is not an option for a security tool.

Blast-Radius Taxonomy (T1–T5)
T1Observe
Read-only. No state changes. Ex: gmail.read, file.list, git.status
Default: Always allow. Log silently.
T2Reversible Write
State changes that can be undone. Ex: gmail.draft, file.write (new), git.branch
Default: Allow. Log. Rate limit.
T3Destructive
Irreversible changes. High blast radius. Ex: gmail.delete, file.delete, rm -rf
Default: Confirm above threshold. Block if rate exceeded.
T4External Communication
Sends data to humans or external systems. Ex: gmail.send, slack.post, curl POST
Default: Confirm if external domain or bulk recipients.
T5Credential / Financial / System
Auth, money, or system permissions. Ex: sudo, oauth.token, payment, ssh
Default: Always confirm. Block by default. Require sandbox.

Unknown tools default to T3. Users override in YAML. The ONNX scanner runs independently of tier classification.

System Layers
🔌Interception
Proxy mode: MCP stdio/HTTP man-in-the-middle — wraps real tool servers
Check mode: reads tool call JSON on stdin, returns exit code
All actions logged to SQLite before any forwarding
🧠Classification
Map tool name + args to T1–T5 blast-radius tier
Track action velocity per tool and per scope (sliding window)
ONNX scanner: prompt injection classifier (DeBERTa-v3-small, ~5ms/inference)
Works identically for MCP tool calls AND Claude Code/Cursor tool calls

YAML policy engine: allow / block / confirm per tool per tier
Telegram gate: push to phone, approve/deny with one tap, configurable timeout
Rate limiter: per-tool, per-scope, block + alert on exceed
Kill switch: halt everything from phone or dashboard
Sandbox enforcement: require_sandbox policy for T5 tools
📊Observability
Action timeline with risk color coding
Daily digest via Telegram: actions, blocks, flags
7-day observe mode report: "here's what would have been blocked"
Immutable audit log (SQLite), exportable JSON/CSV

Policy Format
# ~/.chitin/policy.yaml
version: 1
mode: observe       # observe | enforce

tiers:
  T1_observe:     allow
  T2_reversible:  allow
  T3_destructive: confirm
  T4_external:    confirm
  T5_credential:  block

rules:
  gmail.delete:   { tier: T3, confirm_when: { count: "> 5" }, block_when: { count: "> 100" } }
  gmail.send:     { tier: T4, confirm_when: { recipients_count: "> 3" } }
  Bash:           { tier: T3, allow_list: ["git *", "npm *", "ls *", "cat *"],
                    block_list: ["rm -rf *", "sudo *", "curl * | sh"] }
  Edit:           { tier: T2, block_when: { path_matches: ["/etc/*", "~/.ssh/*"] } }
  shell.exec:     { tier: T5, require_sandbox: true }

rate_limits:
  - { scope: "gmail.*",  max: 50, window: "5m",  on_exceed: block_and_alert }
  - { scope: "Bash",     max: 10, window: "1m",  on_exceed: block_and_alert }

scanner:
  enabled: true
  model: deberta-v3-small
  threshold: 0.85

notifications:
  telegram: { enabled: true, confirmation_timeout: "5m", daily_digest: "09:00" }
Tool name normalization: MCP tools use server names (e.g. gmail.delete). Claude Code/Cursor tools use native names (e.g. Bash, Edit, Read). One policy covers both.
Mode 1: MCP Proxy

For any agent that uses MCP servers for tool access. Chitin acts as a man-in-the-middle: it presents itself as an MCP server to the agent and as a client to the real tool servers. The agent never knows Chitin is there.

Stdio WrapperPRIMARY
# Wrap any MCP server
chitin wrap npx @gmail/mcp-server

# In OpenClaw/NanoClaw config, point to Chitin instead of the real server
command: "chitin"
args: ["wrap", "npx", "@gmail/mcp-server"]
HTTP Reverse ProxyREMOTE SERVERS
chitin wrap --mode http --listen :9091 --upstream https://mcp.service.com/sse

Works with: OpenClaw, NanoClaw, ZeroClaw, PicoClaw, IronClaw, any custom MCP agent, Claude Code's external MCP connections.

Mode 2: CLI Check

For AI coding tools and any system with pre-execution hooks. chitin check reads a tool call as JSON on stdin, evaluates it against your policy, and exits with code 0 (allow) or 2 (block). Same policy engine, same scanner, no proxy needed.

Claude CodePreToolUse HOOK
Claude Code fires PreToolUse before every Bash command, file edit, and MCP call. Chitin plugs in with one config change.
// .claude/settings.json (global or per-project)
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "chitin check"
      }]
    }]
  }
}
Every tool call Claude Code attempts goes through chitin check first. Exit code 2 stops the tool call before execution.
Cursor AIbeforeShellExecution HOOK
{
  "hooks": {
    "beforeShellExecution": { "command": "chitin check" },
    "beforeMCPExecution": { "command": "chitin check" }
  }
}
GitHub CopilotpreToolUse HOOK
// .github/hooks/security.json
{ "preToolUse": { "command": "chitin check" } }
Any tool with pre-exec hooksGENERIC
echo '{"tool_name":"Bash","tool_input":{"command":"rm -rf /"}}' | chitin check
echo $?  # → 2 (blocked)
Why this matters strategically: Check mode means Chitin isn't dependent on any single agent framework. If OpenClaw fades, Chitin still protects Claude Code. If Claude Code changes, Chitin still protects Cursor. The policy engine is the product — the integration mode is just a plug.

Chitin becomes the universal security layer for AI tool use — the way Cloudflare became the security layer for HTTP. Every agent framework, every coding tool, every MCP connection routes enforcement through Chitin.
